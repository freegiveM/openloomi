# Proactive Execution Brain — Decisions Surface Before You Ask

## Four Streams of Inbound + Whatever You Throw At It, One Decision Queue

The average professional has 4 streams of inbound — Gmail, Calendar, GitHub, Slack — and no single place where "what should I do next" lives. You tab-switch all day, re-read the same thread three times, and miss the calendar RSVP until it's already started. Just as often, the real work isn't inbound at all — it's the half-formed thought on your walk, the screenshot of a deadline you noticed in passing, the article you bookmarked for later, the Slack thread someone pinged you about.

OpenLoomi's **Proactive Execution Brain** runs inside the desktop app, watches every channel in the background, enriches every new signal through your local memory, and turns each one into a typed decision card ready for a one-keystroke action — or, when you point it at messy input, a structured daily plan that checks back in.

**What it does:**

| Step            | Action                                                                                                                                                                 |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 👀 **Watch**    | Pulls new signals from Gmail, Calendar, GitHub, and Slack via Composio MCP and accepts voice memos, screenshots, links, pasted notes — every tick, in parallel         |
| 🧠 **Enrich**   | Looks up senders / organizers / repos in your memory so the card arrives with full context attached — and **remembers** new contacts and recurring projects as it goes |
| 🏷️ **Classify** | Maps each signal to a typed decision: `rsvp`, `draft_reply`, `review_pr`, `todo`, `im_reply`                                                                        |
| ✅ **Pick**     | Review the queue in OpenLoomi's Loop panel, the local web UI, or the REPL — one keystroke to execute                                                                   |

### Continuous Background Watch

The Loop is a scheduled task that ticks every 10 minutes by default. Every tick pulls fresh signals, enriches them through your OpenLoomi memory, and appends typed decisions to the queue. Nothing fires automatically — you stay in control of execution.

```text
tick 1  →  Pull (Composio MCP)  →  4 gmail + 1 calendar + 2 PRs + 1 slack
       →  Enrich  →  sender / org / repo lookup against your memory
       →  Filter hard rules  →  noreply@*, Gmail Promotions, already-replied
       →  Classify  →  rsvp ×1 · draft_reply ×2 · review_pr ×1 · im_reply ×1
       →  Queue  →  data/decisions.json (pending bucket)
```

---

### Typed Decisions, Not Raw Noise

The Loop never surfaces raw emails or PRs — every signal becomes a typed decision with a confidence score and the memory refs that explain why it surfaced:

```json
{
  "id": "dec_a1b2c3d4",
  "type": "rsvp",
  "title": "RSVP: Q2 Roadmap Review (Sarah's invite)",
  "confidence": 0.85,
  "memory_refs": ["people/sarah_chen.md", "projects/q2_roadmap.md"],
  "action": {
    "kind": "calendar_rsvp",
    "params": { "event_id": "evt_42", "response": "accepted" }
  }
}
```

`rsvp` from Calendar, `draft_reply` from Gmail, `review_pr` from GitHub — each typed so you know what kind of work it is before you open it.

### A Memory Layer That Reads, Writes, and Evolves With You

The Loop's memory isn't a static profile you set up once and forget. It's a living layer the Loop co-writes with you across three phases, so every decision arrives enriched by what's been learned before — and writes back what was learned now.

**1. Watch + Remember**

Every tick enriches first, then remembers what was just learned:

- Looks up senders, organizers, and repos you've worked with before
- Notes brand-new contacts so they show up by name next time
- Captures recurring calendar event titles as ongoing projects

After a few weeks, the Loop has quietly built a contact and project graph you never had to maintain by hand.

**2. Execute + Remember**

When you pick a card, the Loop carries out the action with the full memory context already loaded. **Any new people or insights the action uncovers are remembered immediately**, so the next decision already knows about them — and the one after that.

**3. Self-Evolve**

Between ticks, your memory layer tunes itself:

- **📊 Hot context stays hot** — frequently used notes surface first
- **🧬 Living connections** — notes accessed close together strengthen their link over time, and fade apart if you stop using them
- **🔄 Daily refresh (24h)** — recalculates what's still relevant and what's drifting
- **🧹 Weekly tidy (7d)** — merges similar notes, archives low-value ones
- **📉 Gentle retention** — long-unused notes fade out so the layer never gets cluttered

The result: every decision the Loop surfaces arrives enriched by a memory layer that's **read before, written after, and self-evolving in between** — never a static snapshot.

### One-Click Execution

Pick a card and the Loop carries out the action with the full context — sender memory, project notes, the original signal, and the suggested action — all loaded in. OpenLoomi captures the result and moves the card to `done` with the action recorded.

The executing session also **remembers any new people or insights it discovers**, so the next tick already knows about them. The memory layer never goes cold between decisions — it grows every time you act.

## How to Set Up

### Step 1: Configure your API keys in OpenLoomi

> **Prerequisite:** Before this step, finish [Getting Started → AI Configuration](/docs/getting-started#ai-configuration). The Loop runs on top of OpenLoomi's AI provider — one setup powers both the chat and the Loop.

Both keys live in **Settings → API Settings**. OpenLoomi uses them in tandem — one for the local server, one for the Composio MCP toolkit the Loop calls into.

| Key                  | Where to get it                                                                            | Why OpenLoomi needs it                                                                          |
| -------------------- | ------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------- |
| **COMPOSIO_API_KEY** | Sign up at [composio.dev](https://composio.dev), then **Settings → API Keys → Create Key** | Authenticates the Composio MCP toolkit that fetches signals from Gmail, Calendar, GitHub, Slack |

> **💡 Prefer the terminal? Use the Composio CLI instead.** Recommended for headless servers, CI, and local debugging — install the CLI, log in, and the Loop picks the key up automatically (no need to paste it in **Settings → API Settings**):
>
> ```bash
> curl -fsSL https://composio.dev/install | bash
> composio login
> ```
>
> The CLI writes the key to `~/.composio/.env`; OpenLoomi's Loop reads it from there on the next tick. Use `composio whoami` to confirm you're authenticated, and `composio logout` to clear it.

After pasting both keys (or after `composio login`), click **Test Connection** in **Settings → API Settings** — OpenLoomi will confirm both endpoints are reachable before you move on.

---

### Step 2: Connect the four toolkits

Each toolkit OpenLoomi exposes for the Loop routes through Composio under the hood. You only need OAuth once. We recommend doing this through the **Composio CLI** (or the `composio-cli` skill from inside OpenLoomi chat) — it handles the OAuth browser flow, persists refresh tokens locally to `~/.composio/`, and is scriptable for headless servers.

1. Log in (skip if you already ran `composio login` in Step 1):

   ```bash
   curl -fsSL https://composio.dev/install | bash
   composio login
   ```

2. Add each of the four toolkits in turn — the CLI opens the OAuth browser flow for each one and writes the resulting refresh token to `~/.composio/`:

   ```bash
   composio add gmail
   composio add googlecalendar
   composio add github
   composio add slack
   ```

   | Toolkit             | CLI command                   | What it powers          | OAuth scope                                    |
   | ------------------- | ----------------------------- | ----------------------- | ---------------------------------------------- |
   | **Gmail**           | `composio add gmail`          | `draft_reply` decisions | Read inbox metadata, send on your behalf       |
   | **Google Calendar** | `composio add googlecalendar` | `rsvp` decisions        | Read events, patch event responses             |
   | **GitHub**          | `composio add github`         | `review_pr` decisions   | Read PRs where you're a reviewer, post reviews |
   | **Slack**           | `composio add slack`          | `im_reply` decisions | Read channels you're in, post replies          |

3. Confirm all four are connected:

   ```bash
   composio connections list
   ```

   Each toolkit should show a status of `ACTIVE`. If any show `INACTIVE` or are missing, re-run `composio add <toolkit>` to redo the OAuth flow. You can also run `composio whoami` at any time to confirm your overall auth state.

> **💡 Prefer to stay inside OpenLoomi?** Invoke the `composio-cli` skill from chat — "connect gmail, googlecalendar, github, and slack for the Loop" — and it drives the same login + OAuth flow without leaving the app. Both the CLI and the skill write to the same `~/.composio/` store the Loop reads from, so they stay in sync.

---

### Step 3: Start the Loop — just ask OpenLoomi

The Loop is a **continuously running execution brain** — it needs a scheduled task to tick on a regular interval. The fastest way to start it is to simply ask OpenLoomi in chat:

```
Start the openloomi-loop proactive execution brain.
Run a tick every 10 minutes so new signals from Gmail / Calendar / GitHub / Slack
get pulled, enriched through your memory, classified into typed decisions
(rsvp / draft_reply / review_pr / todo / im_reply), and queued in the Loop panel.
```

OpenLoomi handles the rest — it creates the scheduled task, registers it in **Agent → Automation**, and runs the first tick immediately so the queue isn't empty. The Loop panel in the sidebar starts showing cards within seconds.

## Why This Works

**🧠 Decisions, Not Streams**

- Four inboxes collapse into one typed queue: `rsvp`, `draft_reply`, `review_pr`, `todo`, `im_reply`, etc.
- Each card carries the memory refs that explain why it surfaced — no guessing, no re-reading
- Confidence score (0.85 known / 0.60 cold) lets you triage in one glance

**⚡ Continuous Without Burnout**

- The Loop ticks every 10 minutes by default; you only open the panel when there is something to do
- Watch timers run independently — a hung tick never blocks notifications on new decisions
- Nothing fires automatically; execution is always an explicit `▶ RUN` click on the card

**🔄 Graceful Degradation**

- Missing Composio toolkit → falls back to your memory layer for that channel's last-known insights
- A lighter memory state → the Loop still produces typed decisions from raw signals alone
- Queue never goes silent; the Loop works on day one with zero toolkits connected

**🌱 Self-Evolving Memory**

- Loop reads and writes your memory layer in the same tick — new senders, projects, and insights all land as soon as they're discovered
- Executing a decision carries out the action and **writes back** any new people or insights it uncovers, so the next tick already knows about them
- Between ticks, your memory self-tunes: hot context stays hot, related notes strengthen their links, a daily refresh recalculates relevance, weekly compaction trims the noise — the layer that grounds every decision is never frozen
