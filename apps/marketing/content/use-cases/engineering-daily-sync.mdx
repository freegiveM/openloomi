# Engineering Daily Sync

## Automated Dev Reports + Issue Sync to Linear — Every Night

Engineering teams waste hours every week pulling commit logs, writing status reports, and triaging GitHub issues. What if every night at 10 PM, AI automatically fetched all commits, compiled a dev report, emailed it to the team, and synced new issues directly to Linear?

**What it does:**

| Step           | Action                                                                                       |
| -------------- | -------------------------------------------------------------------------------------------- |
| 🌙 Every 10 PM | Pull all commits from your GitHub repo                                                       |
| 📝 Compile     | Generate a structured daily dev report                                                       |
| 📧 Email       | Send the report to the engineering team                                                      |
| 🐛 New Issues  | Fetch all issues created that day, rewrite with full detail                                  |
| 🔄 Linear Sync | Sync rewritten issues to Linear for prioritization                                           |
| 🔍 PR Review   | Surface a `review_pr` decision card when a PR is assigned, @-mentions you, or has failing CI |

---

## The Setup

### Configure the PD Prompt

Paste the automation prompt into OpenLoomi. It defines the nightly workflow: commit fetching, report compilation, email delivery, and Linear sync.

---

### Create the Automation Tasks

Two scheduled tasks handle the full workflow:

| Task              | Schedule           | Purpose                                      |
| ----------------- | ------------------ | -------------------------------------------- |
| **PD Daily Sync** | Every day at 10 PM | Fetch commits, compile report, email to team |
| **PD Issue Sync** | Every 1 hour       | Monitor new issues, rewrite, sync to Linear  |

Both tasks are created with tracking enabled for full observability.

> 💡 **PR review isn't in this table** because it's not a scheduled task — it's an event-driven flow inside Loop's tick. The moment a PR is assigned to you or @-mentions you, Loop surfaces a `review_pr` decision card with the AI summary, CI status, and inline comment drafts. No separate task to wire up; just enable the GitHub source in Loop's settings.

## The Operation

### Every 10 PM: PD Daily Sync Runs

OpenLoomi executes the full pipeline — fetching commits, compiling the report, and syncing issues — then generates structured artifacts for each output.

### Execution Complete

After a few minutes, the task finishes with two artifacts: the daily dev report and the issue sync summary.

The sync produces two structured artifacts — the daily dev report and the issue migration summary — each ready to share or archive.

---

### Every Hour: PD Issue Sync Runs

PD Issue Sync doesn't just copy raw issues into Linear — each issue is rewritten with the context the team needs to triage, so the next person who opens the Linear ticket doesn't have to bounce back to GitHub to understand what was reported.

| What Loop pulls                       | What Loop writes into Linear                                                                       |
| ------------------------------------- | -------------------------------------------------------------------------------------------------- |
| Issue title + body                    | Verbatim, prefixed with a `Reported via GitHub on YYYY-MM-DD by @user` header                      |
| Linked PRs / referenced commits       | Auto-resolved and appended as a `Related code` section — the ticket is no longer a black box       |
| Labels + milestone                    | Mapped to Linear's priority / project fields; new labels need your approval before they're created |
| Top 10 comments (by reactions)        | Threaded as a `Context` section so the triage discussion isn't lost                                |
| Reproduction info from issue template | Pulled out into a separate `Steps to reproduce` block in the Linear description                    |

Sync runs are idempotent — if Loop already created a Linear ticket for that GitHub issue (matched by source URL), it updates the existing one instead of duplicating. You can pin a per-repo override list in `~/.openloomi/loop/issue-sync-overrides.json` if a specific repo needs different field mapping.

### When a PR Lands: PD PR Review

PR review isn't a scheduled task — it's an event-driven flow inside Loop's tick. The moment a PR is assigned to you, @-mentions you, or has a CI failure you should look at, Loop surfaces a **`review_pr`** decision card. Same Loop engine, same human-in-the-loop gate, just tuned for code review.

What the card surfaces before you tap **APPROVE**:

| Field on the card        | Source                                                                               |
| ------------------------ | ------------------------------------------------------------------------------------ |
| 🟢 / 🔴 CI status        | GitHub Checks API — lint, typecheck, tests, build                                    |
| 📐 Diff summary          | Loop reads the diff and writes a 3-bullet "what changed" — never the raw patch       |
| 🔗 Linked issue          | `Closes` / `Fixes` / `Refs` parsed from the PR body                                  |
| ⚠️ Risky areas           | Files > 400 lines changed, missing tests, new dependencies → flagged with ⚠          |
| 💬 Inline draft comments | Per-file drafts (style / bug / suggestion); you approve before any comment is posted |
| 🧠 Memory hooks          | Loomi pulls related Obsidian notes / past PRs / RFCs and attaches them as evidence   |

**APPROVE** → comments are posted to GitHub and the PR is left in `Awaiting review` state. **EDIT** → jumps into the chat with the draft loaded; say "drop comment 3, shorten comment 1" and Loop redrafts. **LATER** → parked for your next focus block. **SKIP** → dismissed with a `skippedByUser` audit entry.

PR review is on by default for any repo you've starred or been active on in the last 30 days. To tune which repos / authors / labels qualify, open **Settings → Loop → Sources** — the same panel that controls your email and calendar sources.

## How to Set Up

Two connector calls. The same commands work in OpenLoomi Desktop, the Claude plugin, or the Codex plugin — invoke `openloomi-connectors` in chat:

```
use openloomi-connectors to connect GitHub and Linear
```

GitHub and Linear live in Composio's app catalog, so the skill installs Composio on the fly if you don't have it yet. Verify with `list-accounts` — once both show `active`, Loop picks them up on the next tick and **PD Daily Sync** (10 PM nightly report) + **PD Issue Sync** (hourly issue migration) start firing on their own schedules. No automation tasks to wire up by hand.

> 💡 The flow also lights up the **PR review** path automatically — any repo you've starred or been active on in the last 30 days will surface `review_pr` cards the moment a PR is assigned to you or @-mentions you. Tune the list in **Settings → Loop → Sources**.
