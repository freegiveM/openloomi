# Unified Inbox — Auto-Fetch + Context-Aware Draft Generation

A knowledge worker with too many inboxes has 60 emails, 25 Slack DMs, 14 Telegram messages, and a few iMessage threads stacked up by mid-morning. If three of these stack up in an hour, the context-loading cost exceeds the embarrassment of missing the reply, and shorter, more generic replies start going out just to be done with it. The replies are sent, the relationships thin, and they tell themselves it was fine.

The shortfall isn't discipline. It's the cognitive load of holding 200 ongoing conversations at the same time.

The user doesn't want another chatbot that waits to be asked. They want an AI assistant that genuinely fits a high-volume inbox day: it watches every channel continuously, organizes the thread context, and only finds them when there's a decision that genuinely needs to be made.

## Problem 1: Re-reading every thread before every reply

The painful part of high-volume inbox work isn't the writing — it's the context-loading. The exchange used to go like this:

- Sarah emails about the Q3 OKR status
- Slack is opened to check what was discussed Tuesday
- 40 messages are scrolled back through to remember what was decided
- Notes are checked for "was the Q3 deadline Thursday or Friday?"
- Sarah's email is re-read twice because contradicting something said last week is unacceptable
- Only then is the reply written

If three of these stack up in an hour, the morning is gone. If seven stack up, threads start getting dropped — not on purpose, but because the context-loading cost exceeds the embarrassment of missing the reply.

The reuse cost is real: every reply re-loads the same thread, the same notes, the same prior decisions, before a single new sentence can be written.

## Problem 2: Email and Slack are two different worlds

The same person shows up on every channel, but each channel treats them as its own conversation. Sarah who emails Monday is treated as one contact in Gmail; Sarah who DMs on Slack Thursday is a separate thread in another tool. Each reply requires re-establishing who the person is, what project the message is about, what was already decided last time, and what action is expected now.

Action signals — _"RSVP"_, _"please confirm"_, _"reply"_, _"get back to me"_ — are buried under 200 other messages. By the time "the one that mattered" is reached, half the attention has already been spent on the rest, and the reply either gets rushed or never lands.

## What Loop does

Once the connectors are in and the inbox is wired in OpenLoomi, Loop does three things at once:

**1. Connects every channel into one inbox.** The four (or more) channels actually read get wired up:

```
use openloomi-connectors to connect Gmail
use openloomi-connectors to connect Slack
use openloomi-connectors to connect Telegram
use openloomi-connectors to connect iMessage
```

Gmail and Slack live in Composio's app catalog; the seven native Loomi connectors (Telegram / WhatsApp / iMessage / Feishu / DingTalk / QQ / WeChat) don't need Composio at all. Native and Composio-managed accounts can be mixed freely.

Verification runs with `list-accounts` — once every platform shows `active`, Loop picks them up on the next tick.

**2. Unifies the same contact across channels.** Sarah who emails Monday is the same Sarah who DMs on Slack Thursday — the same `people/sarah_chen.md` profile, the same project note, the same thread cluster. One memory, every card. There's no re-learning her per channel.

**3. Surfaces typed decision cards with grounded drafts.** When an inbound from a contact on the important list arrives with an action word, past the default 4-hour reply window, Loop surfaces a typed decision card. The card carries:

- The **draft reply** — pre-written by the agent, grounded in what Sarah and the user have already discussed
- The **reasoning trace** at the bottom — _"Sarah is bumping the Q3 deck timeline — Thursday is close, she wants it locked today"_ — so Loop's read is visible before commitment
- The **confidence score** (default range 0.60–0.85)
- The four-button tray: **Approve** / **Edit Draft** / **Later** / **Skip**

For email, the card type is **`email_reply`** — signal (gmail), type (email), received time, and the full reasoning trace from the **From Loomi** panel — _why_ Loop thinks this needs attention.

## Loop architecture

To make this work, two kinds of working patterns are configured in OpenLoomi.

**Event-driven:**

| Trigger                                                                | What happens                                                                     |
| ---------------------------------------------------------------------- | -------------------------------------------------------------------------------- |
| Inbound email from important contact + action word + past reply window | `email_reply` card surfaces with the draft grounded in memory; four-gesture flow |
| Inbound DM (Slack / Telegram / Lark / iMessage) needing reply          | `im_reply` card with the channel context and the draft                           |

**Scheduled:**

| Task                      | Schedule          | Purpose                                                          |
| ------------------------- | ----------------- | ---------------------------------------------------------------- |
| **Today's To-Do (Brief)** | Every day at 9 AM | Pulls today's calendar + open decision cards + relevant contacts |
| **Today's Done (Wrap)**   | Every day at 6 PM | Lists what Loop handled for you today + what got parked          |

Together they form a loop that runs continuously across every channel:

```
Continuously watch → Organize thread context → Propose grounded draft → Human decides → Send and remember
```

Loop never sleeps. Brief and Wrap are the bookends; the queue runs in between. Compared to configuring an isolated per-channel workflow, the preference is to have OpenLoomi keep understanding _"which of the conversations actually needs a reply right now."_

> 💡 **Auto-send is not in this loop.** Every `email_reply` and `im_reply` card waits for an explicit **Approve**. Loop never sends without a tap; the dry-run is always shown before the outbound action is taken.

## What changed

| Aspect                 | Before                                                                    | After                                                                          |
| ---------------------- | ------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| Reply review time      | 10–15 min per thread re-reading to recall context                         | Tap a bubble, read the card with reasoning trace, Approve                      |
| Cross-channel context  | Switch tools for Slack vs email vs iMessage; lose context each time       | One unified queue — same `email_reply` / `im_reply` card flow, every channel   |
| Sender awareness       | Re-explain who Sarah is, what project this is, what was decided last time | Memory hooks carry sender profile, project note, prior decisions into the card |
| Control                | Full auto-send (risky) or no AI at all (manual only)                      | Human-in-the-loop dry-run, Approve gate, every single time                     |
| End-of-day mental load | Sift through "did I reply to that?" at 6 PM                               | Wrap recap shows what Loop handled for you today                               |

What OpenLoomi saves isn't just the 20 minutes per reply. More importantly, there's no need to keep switching inboxes, re-reading threads, and worrying about which contact was dropped.

## How to set this up

The "unified inbox" is just **multiple connectors, one per platform**. It works in OpenLoomi Desktop, the Claude plugin, or the Codex plugin — the `openloomi-connectors` skill is invoked in chat, once per account:

```
use openloomi-connectors to connect Gmail
use openloomi-connectors to connect Slack
use openloomi-connectors to connect iMessage
```

The skill installs Composio on the fly the first time a platform that lives in its app catalog is asked for — Gmail and Slack do; the seven **native Loomi connectors** (Telegram / WhatsApp / iMessage / Feishu / DingTalk / QQ / WeChat) don't need Composio at all. Native and Composio-managed accounts can be mixed freely — Loop treats them identically.

Not every platform needs to be connected — only the ones actually read. Toggling a connector off stops the Attention Agent from seeing those signals. Accounts can always be added or removed later with the same `connect` / `disconnect` verbs.

Verification runs with `list-accounts` — once every platform shows `active`, Loop picks them up on the next tick. Loop is on by default at a 10-minute scan; tuning happens in **Settings → Loop → Scan frequency** (5 min for sharper signal, 30 min for a calmer desk). Loomi's bubble cadence is tunable separately in **Settings → Attention Agent**.

> 💡 Once connected, **Loomi can also be chatted with about any message** — a specific thread can be asked about, a long back-and-forth summarized, or details from any message in the unified inbox pulled. Loomi reads from the same memory layer, so the answer is grounded in prior context.

OpenLoomi's audit log records every execution — how many times Loop has polled, which cards surfaced, which Approves were tapped, which drafts were edited. This way inbox triage can be handed off to AI while every action it took remains visible.
