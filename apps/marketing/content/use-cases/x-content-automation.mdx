# X Content Drafting

# I Was Typing Every Post From Scratch

I'm a content creator / community manager who runs an AI-focused X account. By Sunday night, I had to draft 5 posts for the week, reply to whatever came up over the weekend, and somehow do it in a voice that sounded like me and not like an LLM. Most weeks, I ran out of time and the drafts were shorter, more generic, more obviously "a thing a content person wrote on a deadline."

The painful part wasn't the writing — it was the consistency. _"Did I already write a thread about Claude's memory layer? Did I already take a hot take on the latest model release? Is the phrasing I'm using this week the same as the phrasing I used in March?"_ The voice is in my head; it just doesn't transfer to a blank X compose box at 11 PM on a Sunday.

I wasn't short on discipline. I was short on a way to hold my own voice + the topics I've covered + the inbound that needs a reply in my head at the same time.

Then I tried OpenLoomi.

I didn't want another chatbot that waits for me to ask it something. I wanted an AI assistant that genuinely fits a creator's week: it watches X for the signals, drafts in my voice grounded in what I've already posted, and only finds me when there's a decision I genuinely need to make.

> **One rule that doesn't bend:** Loop never auto-sends. That applies to email, to Slack, to Linear, and to X. So an "X automation" use case is really two workflows stitched together: **Loop watches X for signals** (mentions, replies, DMs that need a reply) and surfaces typed decision cards; the **agent + OpenLoomi Memory** draft posts and replies grounded in your past phrasing and what you've already said on the platform. You tap **Approve** on the card, or refine in Chat, then send via the X connector. The whole loop is human-in-the-loop — no autonomous likes, retweets, or replies, ever.

## Problem 1: Sunday night drafting panic

The exchange used to go like this:

- Open X compose, stare at the blank box
- Open my own profile, scroll the last 10 posts, try to remember the phrasing I used
- Open Notion, find my "drafts" doc, see if anything half-finished is still relevant
- Open the model's release notes, summarize a few bullet points
- Try to write a thread that sounds like me and not like a content farm
- Repeat 5 times

By post 4, the voice was usually gone. By post 5, the deadline was past midnight and I was posting things I wasn't proud of.

I realized this is exactly what AI should be doing. The actual hard part of content drafting isn't the writing — it's gathering the context (what I've already said, the voice I've used, the topics I've covered) and reorganizing it into something I can refine in my own phrasing.

So I opened **Chat** with the agent. The agent reads my past posts and replies from `~/.openloomi/data/memory/`, so the draft matches how I actually write. The **`x-twitter-api`** skill is the public packaging for drafting, posting, and scheduling via the X connector.

Try:

- _"Draft a thread about how Loop's decision-card flow works. 5 posts. Reference our last launch thread from March."_
- _"Pull my last 10 posts about AI agent memory, write a follow-up that goes deeper on the forgetting engine."_
- _"Draft a hot take on the latest Claude release in my voice — punchy, not snarky."_

The agent drafts in my voice; I refine, tap **Approve & Post**, and the X connector publishes. For a 5-post week that used to take me 3 hours of blank-page staring, it's now 1 hour of editing.

> 💡 Your past X posts are not in OpenLoomi Memory by default — they live on X. The first time you ask the agent to draft, it may ask you to point it at past posts (paste a few, or share a thread URL). Once it has a sample, Memory carries the voice across sessions.

## Problem 2: Drafts didn't sound like me

The first few times I tried AI drafting tools, the output was technically competent but unmistakably AI. The phrases were too symmetric, the metaphors were too clean, the humor was too safe. The drafts read like drafts; the engagement reflected it.

Loop handles this through the memory layer. The first time I asked the agent to draft, it asked for samples — I pasted 8 past posts, and the agent wrote a voice profile into `~/.openloomi/data/memory/`. From that point on, every new draft reads the profile before it writes, and the voice gets sharper the more I edit and Approve. The **`x-twitter-api`** skill is just the public packaging for the X connector; the voice work happens in the memory layer.

Per [Loop's memory contract](/docs/loop#how-loop-reads-and-writes-memory):

| Loop reads from memory                             | Loop writes after Approve                      |
| -------------------------------------------------- | ---------------------------------------------- |
| Past posts, prior replies, the topics I've covered | Updated thread cluster, refreshed contact note |

I don't curate any of this. A daily analytics refresh surfaces the contacts and topics I actually engage with; a weekly compaction archives stale entries instead of deleting them. The longer I use it, the sharper the voice match gets — and it works the same way across every card type.

## Problem 3: The inbound that needed a reply

The other category that used to eat my day was the inbound. A peer DMs me about a launch I'm working on. A reader mentions me in a thread about a topic I've written about. A potential customer asks a question that's actually a buying signal buried in a public mention. All of these needed a reply; all of them were easy to miss in the X notification firehose.

Loop handles this through the `x_mentions` custom channel and typed decision cards. The first step is registering the channel once:

```
Register a custom channel called "x_mentions" that polls my X mentions every 15 minutes.
Wrap the Composio X toolkit's "list mentions" tool.
```

The skill writes the record to `~/.openloomi/loop/custom-channels.json` and the next Loop tick picks it up. From that moment on, every mention that clears my gates surfaces as a typed card in my queue.

For DMs that need a reply, Loop surfaces an **`im_reply`** card when the message meets all three gates (per [Attention Agent](/docs/attention-agent)):

- The sender is on your **important contacts** list — anyone appearing ≥ 3 times in OpenLoomi Memory is auto-promoted
- The message contains an action word — _"reply"_, _"DM me"_, _"circle back"_
- The message has passed your usual reply window (default 4 hours) and is still unanswered

The card carries the draft reply (grounded in what I and the sender have discussed before, per memory), a reasoning trace, the confidence score, and the four-gesture flow — **Approve / Edit Draft / Later / Skip**. Loomi bubbles the high-confidence ones in the corner of my screen.

For public mentions — `@OpenLoomi your thing is interesting, how does X work?` — Loop surfaces a **`todo`** card with the mention attached. I tap, the agent reads the mention and my prior X voice from memory, drafts a reply in my phrasing, and I send it via the X connector with one tap.

## What I really wanted to solve isn't a few automation scripts

At first I just wanted to save the 3 hours every Sunday I was spending on drafting. But after a few weeks, I realized the real problem isn't that I'm missing a few automation scripts — it's that content work is fundamentally a memory + voice problem:

- My voice lives in 500 past posts, none of which I can re-read before every new draft
- The topics I've covered live in the same posts, distributed across 18 months
- The inbound that needs a reply lives in the X notification firehose
- The voice consistency I'm trying to maintain lives in my head, not in the tools

In the past, all this had to be carried by me actively re-reading. The change OpenLoomi brings is inverting that relationship: instead of me re-reading my own posts to remember the voice, Memory carries it across weeks, and the agent drafts in it.

- If a thread needs writing, the agent reads the prior posts in my voice and drafts the new one.
- If a DM needs a reply, the `im_reply` card lands with the channel context and a grounded draft.
- If a public mention deserves a reply, the `todo` card surfaces with the mention attached.
- If my voice shifts over time, the memory profile shifts with it — the longer I use it, the sharper the match gets.

I only need to focus on the parts that genuinely need my judgement — the actual take, the actual phrasing, the actual judgment about whether to engage.

## From "Sunday night panic" to a continuously running X loop

To make this work, I configured two kinds of working patterns in OpenLoomi.

**One is event-driven:**

| Trigger                                                             | What happens                                                              |
| ------------------------------------------------------------------- | ------------------------------------------------------------------------- |
| Inbound DM from important contact + action word + past reply window | `im_reply` card surfaces with draft grounded in memory; four-gesture flow |
| Public mention on X (via `x_mentions` custom channel)               | `todo` card surfaces with the mention attached                            |

**The other is scheduled:**

| Task                      | Schedule          | Purpose                                                                          |
| ------------------------- | ----------------- | -------------------------------------------------------------------------------- |
| **Brief (Today's To-Do)** | Every day at 9 AM | Pulls today's calendar + open decision cards + overnight X mentions Loop flagged |
| **Wrap (Today's Done)**   | Every day at 6 PM | Lists the replies Loop handled for you today                                     |

Together they form a loop that runs continuously across X and the connected channels:

```
Continuously watch X → Read memory-grounded voice → Propose draft or reply → Human decides → Send via X connector
```

Compared to configuring an isolated per-channel workflow, I'd rather have OpenLoomi keep understanding _"what's the voice I've been writing in, and what does the week need from it."_

> 💡 **Outbound posts aren't in this table** because there's no incoming signal yet. They live in Chat. Open the main OpenLoomi window, ask the agent to draft, refine, then send via the X connector.

## What This Workflow Doesn't Do

Honesty about what Loop isn't, so you set the right expectations:

- ❌ **No autonomous posting.** Loop never publishes on its own. Every post and reply waits for your tap.
- ❌ **No autonomous likes, retweets, or follows.** These aren't supported. Engagement that grows an account has to come from a human (or be planned explicitly in Chat and approved one-by-one).
- ❌ **No "post on a schedule" runner.** If you want to publish at a specific time, you Approve the draft and use the X connector's schedule feature, or set up a manual reminder.
- ❌ **No X growth-hacking automation.** Follow-back loops, auto-DM funnels, engagement pods — these aren't Loop features and run counter to its human-in-the-loop design.

What Loop **does** give you: a watched inbox for X signals, drafts grounded in your voice, and a single tap to publish.

## How to set this up

Two paths, both start from chat.

### Path A: Connect X and register the mentions channel

```
use openloomi-connectors to connect X
Register a custom channel called "x_mentions" that polls my X mentions every 15 minutes.
Wrap the Composio X toolkit's "list mentions" tool.
```

X lives in Composio's app catalog; the `connect` skill installs Composio on the fly if I don't have it yet. Verify with `list-accounts` — once X shows `active` and the `x_mentions` channel appears in chat, Loop picks them up on the next tick.

### Path B: Add the `x-twitter-api` skill for drafting and posting

```
Add the x-twitter-api skill so I can draft and post in chat.
```

This installs the public skill that wraps the X connector for drafting, posting, and scheduling.

### Tune Loop

Loop is on by default at a 10-minute scan. To tune trusted contacts, source toggles, and brief/wrap times, open **Settings → Loop**. To tune when Loomi bubbles appear, open **Settings → Attention Agent**.

> 💡 Your past X posts are not in OpenLoomi Memory by default — they live on X. The first time you ask the agent to draft, it may ask you to point it at past posts (paste a few, or share a thread URL). Once it has a sample, Memory carries the voice across sessions.

OpenLoomi's audit log records every execution — which mentions Loop flagged, which drafts the agent wrote, which Approves I tapped, which posts went out. This way I can hand the voice work off to AI and still see exactly what it did.

## What I need isn't another chatbot

This experience reinforced something for me: an AI content assistant shouldn't be a chat window waiting for me to ask.

If every interaction still requires me to open a tool, type a question, and supply the context, then the responsibility for carrying the voice and watching the inbox still sits on my shoulders.

What I actually need is an AI that can keep working around the content cycle:

- It watches the channels that matter — X, email, Slack, the rest of my stack
- It keeps spotting the inbound that needs a reply
- It proactively organizes the voice and topic context that's scattered across 500 past posts
- It drafts the new post grounded in what I've already written
- It hands the publish button back to me when judgement is needed
- It remembers my edits and choices, gradually learning the voice I actually want
